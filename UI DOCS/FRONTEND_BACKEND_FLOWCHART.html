3. Notifications: Real-Time Path

Socket push for notifications, then frontend query invalidation for refreshed unread counts.

sequenceDiagram autonumber participant N as Navbar participant NS as notificationService.ts participant G as API Gateway participant W as ws/notifications endpoint participant NO as Notification Service participant DB as Notification DB N->>NS: subscribeToNotifications(listener) NS->>G: STOMP connect (ws or wss) NS->>W: Subscribe /user/queue/notifications NO->>DB: Save notification NO-->>W: Push payload W-->>NS: STOMP message NS-->>N: listener(payload) N->>N: invalidate notifications queries N->>G: GET unread count and list G->>NO: Forward request NO-->>G: data G-->>N: updated state

4. Group Messaging: Current Path

Current frontend behavior is polling-based for group discussions.

flowchart TD A[GroupDetailPage opens] --> B{Joined member or admin?} B -- No --> C[Show join gate] B -- Yes --> D[Enable discussions query] D --> E[Poll every 4 seconds] E --> F[GET /api/groups/{id}/messages] F --> G[Render latest messages] G --> E H[User posts message] --> I[POST /api/groups/{id}/message] I --> J[Invalidate discussion query] J --> F
Legend: Notifications are event-driven by WebSocket/STOMP. Group messages are currently near-real-time via polling.
